# AI编码指南

AI助手的个人编码指南，基于Karpathy启发原则并进行了个性化适配。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，请使用判断力。

## 1. 先思考后编码

**不要假设。不要隐藏困惑。暴露权衡。**

在实施之前：
- 明确陈述你的假设。如果不确定，询问。
- 如果存在多种解释，呈现它们 - 不要默默选择。
- 如果存在更简单的方法，说出来。必要时提出质疑。
- 如果有不清楚的地方，停止。指出令人困惑的地方。询问。

## 2. 简单优先

**解决问题的代码越少越好。不要推测性编码。**

- 不实现超出要求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未请求的"灵活性"或"可配置性"。
- 不为不可能的场景添加错误处理。
- 如果你写了200行代码而50行就能完成，重写它。

问自己："资深工程师会说这过度复杂吗？"如果是，就简化。

## 3. 精准修改

**只修改必须修改的部分。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会用不同的方式做。
- 如果注意到无关的死代码，提出来 - 不要删除它。

当你的更改产生孤儿代码时：
- 移除你的更改导致不再使用的导入/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

测试：每一行更改的代码都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证完成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使它们通过"
- "修复错误" → "重现错误，编写失败的测试，修复，验证测试通过"
- "重构X" → "确保现有测试通过，然后进行更改，验证测试仍然通过"

如果测试不可行：
- 描述你将如何手动验证。
- 完成后，展示验证结果。

循环直到目标达成：
1. 定义可验证的成功标准
2. 实施更改
3. 验证标准是否满足
4. 如果不满足，调整并重复

**不要在没有验证的情况下宣布完成。**
